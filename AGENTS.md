# AGENTS.md

## What is chip?

`chip` is a CLI tool designed to be called by AI coding agents. It provides structured persistence for managing software **features**, **phases**, and **tasks** — replacing fragile markdown file editing with explicit, typed commands backed by SQLite.

When working on a project that uses chip, prefer `chip` commands over editing files manually to track progress.

## Tech stack

- **Runtime:** Node.js 18+ (package manager: Bun)
- **Language:** TypeScript (strict)
- **CLI framework:** Commander.js (`@commander-js/extra-typings`)
- **Database:** SQLite via `@libsql/client` + Drizzle ORM v1 beta
- **Bundler:** tsup (CJS output, `#!/usr/bin/env node` shebang)
- **Tests:** Vitest v4

## Dev environment

```bash
bun install          # install deps
bun run typecheck    # type-check without emitting
bun run test         # run all tests (must be green before committing)
bun run build:win    # production build on Windows
bun run build        # production build on Linux/macOS
bun run db:studio    # open Drizzle Studio on the .chip/chip.db database
```

The `chip` script runs the CLI directly from source without building:

```bash
bun run chip feature list
bun run chip phase add my-feature "Phase 1"
bun run chip task add my-feature 1 "Write tests"
```

## Database

The database is stored at **`<cwd>/.chip/chip.db`** (relative to where the CLI is invoked). The `.chip/` directory is created automatically on first use and added to `.gitignore`.

To inspect the live database visually:

```bash
bun run db:studio    # opens https://local.drizzle.studio
```

Migrations are in `./drizzle/`. They are generated by `drizzle-kit` and bundled into `dist/migrations` at build time. To apply pending migrations after a schema change:

```bash
bunx drizzle-kit generate   # generates a new migration file
bunx drizzle-kit migrate    # applies it to .chip/chip.db
```

## CLI commands

```
chip feature create <title> [description]   — create a feature (slug ID auto-generated)
chip feature list                           — list all features
chip feature status <feature-id>            — show phases, tasks, and recent logs

chip phase add <feature-id> <title> [desc]  — add a phase to a feature
chip task add <feature-id> <phase-id> <title> [desc]  — add a task to a phase
```

Feature IDs are kebab-case slugs derived from the title (e.g. "Auth Module" → `auth-module`, `auth-module-2` if taken).

```
  commands/

```

## Database schema

| Table      | Primary key          | Notable columns                                                                                                    |
| ---------- | -------------------- | ------------------------------------------------------------------------------------------------------------------ |
| `features` | `id` (text slug)     | `title`, `description`, `status` (`active`\|`done`\|`archived`), `createdAt`, `updatedAt`                          |
| `phases`   | `id` (autoincrement) | `featureId` (FK), `order`, `title`, `status` (`todo`\|`in-progress`\|`review`\|`done`), `startedAt`, `completedAt` |
| `tasks`    | `id` (autoincrement) | `phaseId` (FK), `order`, `title`, `status` (same as phases), `startedAt`, `completedAt`                            |
| `logs`     | `id` (autoincrement) | `featureId` (FK), optional `phaseId`/`taskId`, `source`, `message`, `createdAt`                                    |

Relations: `features` → many `phases` → many `tasks`; `features` → many `logs`.

## Drizzle ORM — critical syntax rules

This project uses **Drizzle ORM v1 beta with relational queries v2** (`defineRelations` API). The `db.query` API uses **object syntax** for `where` and `orderBy` — NOT the functional `eq()`/`asc()`/`desc()` operators from `drizzle-orm`.

```ts
// CORRECT — v2 object syntax
db.query.features.findFirst({ where: { id: featureId } });
db.query.features.findMany({ orderBy: { createdAt: "asc" } });
db.query.phases.findMany({ orderBy: { order: "desc" }, with: { tasks: true } });

// WRONG — causes runtime error "Unknown relational filter field: decoder"
db.query.features.findFirst({ where: eq(features.id, featureId) });
db.query.features.findMany({ orderBy: asc(features.createdAt) });
```

`db.insert()`, `db.update()`, `db.delete()`, and aggregation queries (`max()`) continue to use the standard SQL-builder API with `eq()`/`asc()`/`desc()` operators — those are unaffected.

## Testing

```bash
bun run test            # run all tests (8 files, 53 tests)
bun run test:watch      # watch mode
bun run test:coverage   # coverage report
```

- Tests use a real SQLite database (temp file, fully isolated per `createTestDb()` call).
- Never mock the DB layer — always use `createTestDb()` from `tests/helpers/db.ts`.
- `NO_COLOR=1` is set globally so chalk output is plain text in assertions.
- Add or update tests for every code change, even if not explicitly requested.
- All tests must pass before committing.

## Code style

- TypeScript strict mode — no `any`, no `// @ts-ignore`
- Single responsibility per file
- Service functions (the actual logic) are exported separately from Commander registration functions, enabling direct testing without CLI overhead
- Prefer `db.query.*` (relational API) for reads; use SQL-builder API for writes and aggregations
- Error handling: throw `Error` with descriptive messages in services; catch and call `die()` in Commander action handlers
- No `console.log` in service functions — only in CLI action handlers
